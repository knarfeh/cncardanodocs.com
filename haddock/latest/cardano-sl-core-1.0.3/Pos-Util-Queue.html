Pos.Util.Queue

Description

This impl is ok and is O(1) amortised time provided that the use of dequeue is non-persistent style, meaning that we never call dequeue twice on the same value. In the case of PriorityLock, where this is intended to be used, that's true because it will be the unlocking thread that does the dequeue.

An alternative impl that would be suitable in other cases would do the reversing on the enqueue side.
